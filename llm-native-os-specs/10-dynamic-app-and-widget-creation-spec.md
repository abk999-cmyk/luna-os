# Document 10: Dynamic App and Widget Creation Specification
**LLM-Native Operating System**
**Status:** Implementation Guide
**Version:** 1.0

---

## Executive Summary

This document specifies how the LLM generates custom applications, widgets, and visual interfaces at runtime. This capability—immediate, bespoke UI creation without compilation or deployment—is the defining differentiator of the LLM-native OS. The LLM emits declarative specifications, the OS renders them instantly, the user interacts, and events flow back to the controlling agent.

**The Killer Feature:** Users get hyper-curated, task-specific interfaces that disappear when the work is done. No pre-built, one-size-fits-all UI. The interface evolves with the conversation.

---

## 1. Core Principles

### 1.1 Real-Time Rendering
- **No compilation, no build step.** The LLM emits a declarative spec; the OS renders it in <100ms.
- **No deployment latency.** UI appears instantly in a new window.
- **60fps interaction.** Events generated by user actions must flow back to the agent with imperceptible latency.

### 1.2 Progressive Disclosure by LLM Judgment
- The LLM is **opinionated and smart** about what the user needs now versus later.
- **Start simple.** A travel planner doesn't need a 50-widget dashboard; it needs a map and a day list.
- **Add on request.** User says "add a budget tracker"—LLM revises the spec, OS renders the update.
- **Not minimal.** Provide enough to be useful. Not minimal to the point of uselessness.
- **Not overdone.** Every widget should serve a clear purpose in the current task.

### 1.3 Composability
- UIs are built from **high-level primitives** (DataTable, InteractiveMap, Timeline, etc.), not raw HTML/CSS.
- LLM composes primitives declaratively.
- Primitives handle layout, state, events, and interaction semantics internally.
- Extensible: New primitives can be added without changing the core rendering engine.

### 1.4 Ephemeral by Default
- Dynamic apps exist for the duration of the task.
- When the work is done, the app is dissolved (cleaned up).
- If the user keeps returning to a dynamic app, its actions can be **promoted to persistent** in the action registry (tier 3).
- The entire app can be **saved as a reusable template** for future use.

## 2. Declarative UI Specification Format

### 2.1 Top-Level Structure

All dynamic UIs are described in a **JSON-based declarative format**. The root object defines the app or widget.

```json
{
  "version": "1.0",
  "type": "application",
  "id": "travel-planner-task-123",
  "title": "Japan Trip Planner",
  "description": "Interactive itinerary builder with map and budget tracker",
  "layout": "vertical",
  "width": 1400,
  "height": 900,
  "components": [
    { ... }
  ],
  "actions": [ ... ],
  "styles": { ... }
}
```

### 2.2 Component Schema

Each component in the `components` array follows this schema:

```json
{
  "id": "unique-component-id",
  "type": "ComponentType",
  "props": {
    "key": "value",
    ...
  },
  "children": [ ... ],
  "events": {
    "onSelect": "agent-action",
    "onChange": "agent-action",
    ...
  },
  "layout": {
    "grow": 1,
    "shrink": 1,
    "basis": "auto",
    "align": "center"
  }
}
```

**Fields:**
- **id:** Unique identifier (string). Used to reference component state and receive events.
- **type:** Component type (e.g., "DataTable", "InteractiveMap", "Timeline").
- **props:** Component-specific properties (data, headers, markers, etc.).
- **children:** Nested components (hierarchy).
- **events:** Event handlers. Values are agent action names (strings). Fired when user interacts.
- **layout:** Flexbox-like layout properties (grow, shrink, basis, align, direction).

### 2.3 Component Hierarchy and Layout

Components can be nested. Layout follows a **flexbox model**:

```json
{
  "type": "Container",
  "id": "main-layout",
  "layout": {
    "direction": "row",
    "gap": 16,
    "align": "stretch"
  },
  "children": [
    {
      "type": "Container",
      "id": "left-panel",
      "layout": { "direction": "column", "grow": 1 },
      "children": [
        { "type": "InteractiveMap", "id": "japan-map", ... },
        { "type": "Legend", "id": "map-legend", ... }
      ]
    },
    {
      "type": "Container",
      "id": "right-panel",
      "layout": { "direction": "column", "grow": 1, "gap": 12 },
      "children": [
        { "type": "Timeline", "id": "itinerary", ... },
        { "type": "BudgetTracker", "id": "budget", ... }
      ]
    }
  ]
}
```

### 2.4 Data Binding

Props can reference dynamic data. Data is provided at render time via a **data context**:

```json
{
  "type": "DataTable",
  "id": "destinations-table",
  "props": {
    "headers": ["City", "Days", "Budget"],
    "rows": "$.destinations",
    "sortable": true,
    "selectable": true
  }
}
```

The string `"$.destinations"` is a **JSONPath reference** to data provided in the render context. The OS resolver substitutes actual data at render time.

## 3. Component Primitive Library

### 3.1 Library Overview

The OS provides a curated set of **high-level primitives**. The LLM composes these to build UIs. Each primitive is fully featured and handles its own rendering, state management, and interaction.

#### Primitive Categories:
1. **Data Display:** DataTable, List, Card, Stat, Timeline
2. **Input:** TextInput, NumberInput, DateInput, Select, MultiSelect, Checkbox, Toggle, Slider, ColorPicker
3. **Visualization:** InteractiveMap, Chart, Graph, Canvas, Gauge
4. **Communication:** Chat, Panel, Modal, Toast, Badge
5. **Navigation:** Tabs, Breadcrumbs, Sidebar, Menu
6. **Containers:** Container, Divider, Spacer, Grid
7. **Code:** CodeEditor, CodeBlock, Terminal
8. **Media:** Image, Video, Audio, FilePreview

### 3.2 Primitive Specifications

#### **DataTable**
```json
{
  "type": "DataTable",
  "id": "example-table",
  "props": {
    "headers": ["Name", "Email", "Status"],
    "rows": "$.items",
    "sortable": true,
    "searchable": true,
    "selectable": true,
    "pagination": { "pageSize": 20 },
    "density": "normal"
  },
  "events": {
    "onRowSelect": "handle_row_selection",
    "onSort": "handle_sort",
    "onSearch": "handle_search"
  }
}
```

**Props:**
- `headers` (string[]): Column headers.
- `rows` (JSONPath string): Data rows.
- `sortable` (bool): Enable column sorting.
- `searchable` (bool): Enable row search.
- `selectable` (bool): Enable row selection.
- `pagination` (obj): Pagination config { pageSize, currentPage }.
- `density` (enum): "compact" | "normal" | "spacious".

**Events:**
- `onRowSelect({ rowIndex, rowData })`: Fires when user clicks a row.
- `onSort({ column, direction })`: Fires when user sorts.
- `onSearch({ query })`: Fires when user enters search text.

## 4. Rendering Pipeline

### 4.1 Render Flow

```
LLM emits declarative spec (JSON)
          ↓
OS parses and validates schema
          ↓
OS resolves data references (JSONPath)
          ↓
OS instantiates primitive components
          ↓
OS applies layout engine (flexbox)
          ↓
OS renders to GPU (WebGL/Metal)
          ↓
Display in new window (<100ms from spec emission)
          ↓
User interacts
          ↓
Event generated and sent to agent
```

### 4.2 Validation

Before rendering, the OS validates the spec:

1. **Schema validation:** Spec conforms to JSON schema.
2. **Component type validation:** All component types exist in the primitive library.
3. **Data reference validation:** JSONPath expressions are syntactically valid.
4. **Event validation:** Event handler names are non-empty strings.
5. **Layout validation:** Layout properties are valid.

If validation fails, the OS returns an error to the agent with details. The agent can correct and re-emit.

### 4.3 Data Resolution

Data references use **JSONPath syntax**:

```
$.field              → root.field
$.array[0]           → root.array[0]
$.items[*].name      → all names in root.items
$.deeply.nested.val  → nested access
```

The OS resolves these against a **data context** provided at render time. The data context is supplied by the agent.

### 4.4 Layout Engine

The OS uses a **flexbox-like layout model**:

- `direction` (enum): "row" | "column" (default: "row").
- `gap` (number): Space between children.
- `grow` (number): Flex grow factor.
- `shrink` (number): Flex shrink factor.
- `basis` (string): Flex basis ("auto", "100%", "500px", etc.).
- `align` (enum): "start" | "center" | "end" | "stretch".
- `justify` (enum): "start" | "center" | "end" | "space-between" | "space-around".

Layout is computed once at render and re-computed on window resize. State changes do NOT trigger re-layout; only data and prop updates do.

### 4.5 Rendering Engine

The OS renders components to a **native graphics surface**:

- **Web:** Canvas API + WebGL (for maps, charts, large tables).
- **Desktop (Electron/Tauri):** Native rendering API + Canvas.
- **Mobile:** Native platform APIs.

Rendering is **hardware-accelerated**. Target: **60 fps** for all interactions (scrolling, panning, filtering, etc.).

### 4.6 Performance Targets

- **Render latency:** <100ms from spec emission to pixels on screen.
- **Event latency:** User action → OS event dispatch → agent receives event: <50ms.
- **Interaction frame rate:** 60 fps for scrolling, panning, zooming, and other continuous interactions.
- **Data update latency:** Data change → component re-render: <16ms (one frame at 60fps).

## 5. Event Flow and Agent Communication

### 5.1 Event Emission

When the user interacts with a component, the OS generates an **event object**:

```json
{
  "eventType": "onRowSelect",
  "componentId": "destinations-table",
  "appId": "travel-planner-task-123",
  "timestamp": 1711213567000,
  "data": {
    "rowIndex": 3,
    "rowData": {
      "city": "Kyoto",
      "days": 3,
      "budget": 800
    }
  }
}
```

### 5.2 Event Handler Mapping

In the component spec, the `events` object maps event types to **agent action names**:

```json
"events": {
  "onRowSelect": "handle_row_selection",
  "onSort": "handle_sort"
}
```

When an event fires, the OS:

1. Looks up the handler name (e.g., "handle_row_selection").
2. Dispatches the event to the controlling **agent** (the LLM session managing this app).
3. The agent receives the event as a tool call or function invocation.
4. The agent processes the event, may update data, and may emit a new spec to re-render.

### 5.3 Agent Action Interface

The agent receives events as structured calls:

```
Agent receives: (action_name, event_data)
  e.g., ("handle_row_selection", { rowIndex: 3, rowData: {...} })

Agent responds by:
  - Updating internal state/data
  - Emitting a new UI spec (if UI needs to change)
  - Calling external tools (APIs, databases, etc.)
  - Sending messages to the user
```

### 5.4 Bidirectional Communication

```
Agent → OS: emit_ui_spec(spec, data_context)
OS → Agent: dispatch_event(event_name, event_data)
Agent → OS: emit_ui_spec(updated_spec, updated_data_context)
... (cycle continues)
```

This is **low-latency, event-driven communication**. No polling. Events are prioritized.

## 6. Action Registration from Dynamic Apps

### 6.1 Ephemeral vs. Persistent Actions

Dynamic apps can register **new actions** into the system:

- **Ephemeral actions:** Exist only while the app is running. Destroyed when the app is dissolved.
- **Persistent actions:** Registered into the action registry (tier 3) and available to other agents and future sessions.

### 6.2 Action Registration Spec

When the LLM creates a dynamic app, it can register actions:

```json
{
  "action": "register_dynamic_action",
  "appId": "travel-planner-task-123",
  "actions": [
    {
      "name": "travel.add_destination",
      "description": "Add a new destination to the itinerary",
      "parameters": {
        "city": { "type": "string", "description": "City name" },
        "days": { "type": "number", "description": "Duration" },
        "budget": { "type": "number", "description": "Budget in USD" }
      },
      "persistence": "ephemeral"
    },
    {
      "name": "travel.calculate_budget",
      "description": "Calculate total trip budget",
      "parameters": {
        "items": { "type": "array", "description": "Budget items" }
      },
      "persistence": "ephemeral"
    }
  ]
}
```

### 6.3 Promotion to Persistent

If a dynamic action proves useful (user keeps using it, or agent recommends it), it can be promoted:

```json
{
  "action": "promote_action_to_persistent",
  "actionName": "travel.add_destination",
  "appId": "travel-planner-task-123"
}
```

Promotion moves the action to the **persistent action registry** (tier 3). It becomes available to all agents.

### 6.4 Action Lifecycle

```
Dynamic app created
  ↓
Agent registers ephemeral actions
  ↓
User/agent interacts with actions
  ↓
(Decision: keep or discard?)
  ↓
If keep: Promote to persistent
If discard: App dissolved, actions deleted
```

## 7. Persistence and Templates

### 7.1 Ephemeral Lifecycle

By default, dynamic apps are **ephemeral**:

1. Created for the current task.
2. Live in memory during the agent session.
3. Destroyed when the task completes or the agent session ends.
4. No trace left (unless explicitly saved).

### 7.2 Saving as Reusable Template

If the user (or agent) wants to keep a dynamic app, it can be **saved as a template**:

```json
{
  "action": "save_app_as_template",
  "appId": "travel-planner-task-123",
  "templateName": "Travel Planner",
  "templateDescription": "Interactive trip planning interface with map, itinerary, and budget tracking",
  "category": "travel",
  "tags": ["travel", "planning", "itinerary"]
}
```

The template is saved to persistent storage (filesystem, database, etc.). Future agents can:

```json
{
  "action": "create_app_from_template",
  "templateName": "Travel Planner",
  "dataContext": { ... }
}
```

This instantiates a new app with the same structure but fresh data.

### 7.3 Template Schema

A template is a **parameterized spec**:

```json
{
  "type": "template",
  "name": "Travel Planner",
  "version": "1.0",
  "parameters": [
    {
      "name": "destination",
      "type": "string",
      "description": "Trip destination country/region"
    },
    {
      "name": "duration_days",
      "type": "number",
      "description": "Total trip duration"
    },
    {
      "name": "budget_usd",
      "type": "number",
      "description": "Total budget"
    }
  ],
  "spec": { ... }
}
```

When instantiating, the agent provides parameter values.

### 7.4 Template Discovery

The OS maintains a **template registry**. Agents can query:

```json
{
  "action": "list_templates",
  "category": "travel",
  "tags": ["planning"]
}
```

Returns available templates matching the criteria.

## 8. Smart Middle Ground Design Heuristics

The LLM should follow these heuristics when designing dynamic UIs:

### 8.1 Principle: Start Simple, Expand on Request

**Rule:** Provide the minimum viable set of components to complete the task. Add more on user request or as the task evolves.

**Example: Travel Planner**
- **Initial spec:** Map + day-by-day timeline + basic itinerary input.
- **User asks "show me the budget":** Add budget tracker.
- **User asks "what's the weather?":** Add weather widget.
- **User asks "help me pack":** Add packing list.

**Anti-pattern:** Emit a 50-widget dashboard on the first spec. The user is overwhelmed; the interface is not task-curated.

### 8.2 Principle: Opinionated Layout

**Rule:** The LLM should make smart decisions about layout based on the task. Don't ask the user "do you want a grid or a list?"—choose the better option.

**Example:**
- **Browsing data:** DataTable (sortable, searchable, pagination).
- **Comparing options:** Side-by-side cards.
- **Following a process:** Vertical timeline or wizard.
- **Exploring spatial data:** Map.

### 8.3 Principle: Relevant Context Only

**Rule:** Display only the data and controls relevant to the current step of the task.

**Example:** If the user is selecting a destination, show the map and a list of candidates. Don't show the budget tracker yet; they're not thinking about money yet.

### 8.4 Principle: Progressive Complexity

**Rule:** As the user drills down or advances in the task, add detail and complexity.

**Example:**
- User selects a city → show detailed info (hotels, restaurants, attractions).
- User selects a hotel → show booking interface.
- User books → show confirmation and add-ons.

### 8.5 Principle: Feedback and Validation

**Rule:** Provide immediate visual feedback for user actions. Disable buttons that can't be used. Show validation errors inline.

**Example:**
- Hover over a destination → highlight it on the map.
- Click "Add Day" → new day appears immediately.
- Enter invalid date → error message inline, button disabled.

### 8.6 Principle: Searchability and Filtering

**Rule:** If data can be large, enable search and filtering out of the gate.

**Example:**
- Table of 1000 destinations → include a search box.
- Map with 50 markers → include a filter by city/region.

### 8.7 Principle: Clarity on State

**Rule:** Always show what's selected, what's in progress, and what's complete.

**Example:**
- Selected destination highlighted (color, checkmark, etc.).
- In-progress days marked as "editing".
- Completed days checked or collapsed.

## 11. API Reference

### 11.1 Agent API

#### `emit_ui_spec(spec, data_context)`
Emits a declarative UI spec with data context.

**Parameters:**
- `spec` (object): Declarative UI specification (JSON schema).
- `data_context` (object): Data context for JSONPath resolution.

**Returns:** `{ appId, status }`

---

#### `dispatch_event(event_name, event_data)`
Dispatches an event from the OS to the agent.

**Parameters:**
- `event_name` (string): Handler name (e.g., "handle_row_selection").
- `event_data` (object): Event details (varies by event type).

**Called by:** OS (agent receives this as an invocation).

---

#### `register_dynamic_action(appId, actions)`
Registers ephemeral actions for a dynamic app.

**Parameters:**
- `appId` (string): App ID.
- `actions` (array): Action specifications.

**Returns:** `{ registered: [action names], errors: [...] }`

---

#### `promote_action_to_persistent(actionName, appId)`
Promotes an ephemeral action to persistent.

**Parameters:**
- `actionName` (string): Action name.
- `appId` (string): App ID.

**Returns:** `{ status: "promoted" }`

---

#### `save_app_as_template(appId, templateName, templateDescription, category, tags)`
Saves a dynamic app as a reusable template.

**Parameters:**
- `appId` (string): App ID.
- `templateName` (string): Template name.
- `templateDescription` (string): Description.
- `category` (string): Category (e.g., "travel").
- `tags` (array): Tags for discovery.

**Returns:** `{ templateId, status }`

---

#### `create_app_from_template(templateName, dataContext)`
Creates a new dynamic app from a saved template.

**Parameters:**
- `templateName` (string): Template name.
- `dataContext` (object): Data to inject.

**Returns:** `{ appId, spec, status }`

---

#### `dissolve_app(appId)`
Destroys a dynamic app and cleans up resources.

**Parameters:**
- `appId` (string): App ID.

**Returns:** `{ status: "dissolved" }`

---

### 11.2 Primitive Component Props (Common)

All primitives accept:
- `id` (string): Unique component ID.
- `props` (object): Component-specific properties.
- `layout` (object): Layout properties (direction, gap, grow, shrink, basis, align, justify).
- `events` (object): Event handler mappings.
- `children` (array): Nested components.

## 12. Constraints and Limitations

### 12.1 Known Constraints

1. **Rendering latency:** Spec emission to pixels: <100ms. If data is very large (10M+ rows), pre-filter/paginate on agent side.
2. **Component complexity:** Primitives are composable. Complex custom logic should be in the agent, not in primitives.
3. **State management:** Component state is ephemeral. Persistent state must be in the agent.
4. **Concurrent edits:** Multiple agents manipulating the same app spec are not supported. Apps are owned by one agent.
5. **Network:** Event latency assumes local or low-latency network (LAN, cloud co-location). WAN delays may exceed targets.

### 12.2 Future Extensions

1. **Collaborative editing:** Multi-agent UI co-creation (future version).
2. **Custom renderers:** Allow agents to register custom rendering logic (future version).
3. **Animation system:** Built-in transitions and animations (future version).
4. **Accessibility:** WCAG compliance and screen reader support (future version).
5. **Dark mode:** Automatic dark theme support (future version).

## 13. Conclusion

Dynamic app and widget creation is the defining feature of the LLM-native OS. By emitting declarative specs, the LLM can create task-curated interfaces instantly without any build or deployment process. The rendering pipeline is optimized for performance (<100ms latency, 60 fps interaction). Events flow back to the agent in real time. Ephemeral apps can be saved as templates or dissolved when work is done. Actions registered by dynamic apps can be promoted to persistent.

The smart middle ground heuristics ensure that UIs are useful, opinionated, and not overwhelming. Progressive disclosure, relevant context, and feedback make the experience feel natural and responsive.

This capability fundamentally changes how humans and AI agents interact with systems. Instead of navigating fixed menus and forms, users get interfaces that adapt to the task at hand.
